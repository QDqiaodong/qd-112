# 工具详情页与保养周期修复说明

## 一、问题概述

本次修复解决了两个核心问题：

1. **工具详情页 API 缺失问题**：前端工具详情页调用了两个不存在的后端接口，导致保养记录和使用记录无法加载
2. **保养周期计算逻辑缺陷**：保养后下次保养日期仅按工具自身 `maintenanceCycleDays` 计算，未接入维护类型（MaintenanceItem）对应的独立周期配置

---

## 二、问题一：工具详情页 API 缺失

### 2.1 现象

工具详情页（ToolDetail.vue）中以下数据加载失败：
- **保养记录 Tab**：调用 `GET /api/tools/{id}/maintenance` 接口，返回 404
- **使用记录 Tab**：调用 `GET /api/tools/{id}/usage` 接口，返回 404

### 2.2 根因分析

前端 API 定义如下（按工具维度的 RESTful 设计）：

| 文件 | 函数 | 调用路径 |
|------|------|----------|
| `frontend/src/api/maintenance.ts` | `getMaintenanceByTool()` | `GET /tools/{toolId}/maintenance` |
| `frontend/src/api/usage.ts` | `getUsageByTool()` | `GET /tools/{toolId}/usage` |

而后端 `ToolController` 仅提供了以下相关端点，**缺少按工具 ID 查询保养/使用记录的端点**：

| 现有端点 | 说明 |
|----------|------|
| `GET /api/maintenance?toolId=xxx` | 保养记录列表（带查询参数） |
| `GET /api/maintenance/tool/{toolId}` | 保养记录（不分页） |
| `GET /api/usage?toolId=xxx` | 使用记录列表（带查询参数） |
| `GET /api/usage/tool/{toolId}` | 使用记录（不分页） |

### 2.3 修复方案

在 `ToolController` 中新增两个 RESTful 端点，内部复用已有的 Service 层分页查询能力：

**新增端点一：`GET /api/tools/{id}/maintenance`**
- 路径参数：`id` - 工具 ID
- 查询参数：`page`（默认 1）、`size`（默认 10）
- 返回：`ApiResponse<PageResult<MaintenanceRecord>>`

**新增端点二：`GET /api/tools/{id}/usage`**
- 路径参数：`id` - 工具 ID
- 查询参数：`page`（默认 1）、`size`（默认 10）
- 返回：`ApiResponse<PageResult<UsageRecord>>`

### 2.4 修改文件

| 文件 | 变更说明 |
|------|----------|
| `backend/src/main/java/com/home/tools/controller/ToolController.java` | 注入 `MaintenanceRecordService`、`UsageRecordService`，新增两个 `@GetMapping` 端点 |

## 三、问题二：保养后下次日期回写未接入维护类型周期

### 3.1 现象

创建或更新保养记录时，下次保养日期（`nextMaintenanceDate`）仅按工具表的 `maintenanceCycleDays` 字段计算，未考虑不同维护类型（清洁、润滑、紧固、检查等）各自的独立周期配置。

例如：清洁的周期应为 30 天，但实际却按工具的通用周期 90 天计算。

### 3.2 根因分析

**现有逻辑**（MaintenanceRecordServiceImpl.create/update）：
```java
if (record.getNextMaintenanceDate() == null && tool.getMaintenanceCycleDays() != null) {
    record.setNextMaintenanceDate(
        record.getMaintenanceDate().plusDays(tool.getMaintenanceCycleDays())
    );
}
```

**问题**：完全忽略了 `maintenance_item` 表中配置的各维护类型独立周期 `default_cycle_days`。

### 3.3 维护类型周期配置（已有数据）

系统已在 `maintenance_item` 表中预置了维护类型及其默认周期（通过 `DataInitializer` 初始化）：

| 维护项 code | 维护项名称 | 默认周期（天） | 对应 MaintenanceType |
|------------|-----------|--------------|---------------------|
| clean | 清洁除尘 | 30 | CLEAN |
| oil | 润滑上油 | 60 | OIL |
| tighten | 螺丝紧固 | 90 | TIGHTEN |
| inspect | 零件检查 | 180 | INSPECT |
| battery | 电池维护 | 30 | - |
| cable | 电线检查 | 90 | - |
| sharpen | 刀具打磨 | 60 | - |
| rustproof | 防锈处理 | 180 | - |

> 注：`MaintenanceType` 枚举值转为 **小写** 后与 `maintenance_item.code` 进行匹配。
> 对于未匹配到的维护类型（如 `REPAIR`、`OTHER`），回退使用工具的通用周期。

### 3.4 修复方案

提取独立的周期解析方法 `resolveNextMaintenanceDate()`，优先级如下：

```
用户手动指定 nextMaintenanceDate 
    → 直接使用，不覆盖
否则，按维护类型（maintenanceType）查找 MaintenanceItem.defaultCycleDays
    → 找到则使用该周期
否则，回退使用 Tool.maintenanceCycleDays
    → 找到则使用该周期
否则，不设置 nextMaintenanceDate（留空）
```

### 3.5 修改文件

| 文件 | 变更说明 |
|------|----------|
| `backend/src/main/java/com/home/tools/repository/MaintenanceItemRepository.java` | 新增 `findByCode(String code)` 方法，支持按 code 查询维护项 |
| `backend/src/main/java/com/home/tools/service/impl/MaintenanceRecordServiceImpl.java` | 1. 注入 `MaintenanceItemRepository`<br>2. 新增 `resolveNextMaintenanceDate()` 方法<br>3. `create()` 和 `update()` 中调用新方法替代原有硬编码逻辑 |

## 五、验证方式

### 5.1 验证 API 端点

启动后端服务后，使用以下命令验证：

```bash
# 验证保养记录接口（假设工具 ID 为 1）
curl http://localhost:8080/api/tools/1/maintenance?page=1&size=20

# 验证使用记录接口
curl http://localhost:8080/api/tools/1/usage?page=1&size=20
```

预期：返回 200 状态码，响应体中包含 `data.list` 数组结构。

### 5.2 验证保养周期计算

创建一条"清洁"类型的保养记录，预期下次保养日期 = 保养日期 + 30 天（而非工具的通用周期）：

```bash
curl -X POST http://localhost:8080/api/maintenance \
  -H "Content-Type: application/json" \
  -d '{
    "toolId": 1,
    "maintenanceType": "CLEAN",
    "maintenanceDate": "2026-06-15",
    "operator": "测试员",
    "description": "清洁测试"
  }'
```

预期：返回的 `nextMaintenanceDate` 为 `2026-07-15`（30 天后）。
